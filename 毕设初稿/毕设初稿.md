Design of a Child-Oriented Intelligent Robot Dog and Research on AI Interaction Functions

Abstract：
Child-oriented companion robots require both expressive motions and natural voice interaction. However, deploying speech pipelines and large-model inference directly on resource-constrained embedded controllers is challenging due to computation, power, and real-time requirements. This work proposes and implements an intelligent quadruped robot dog prototype with a cooperative architecture consisting of a real-time motion-control terminal and an independent voice-AI interaction terminal. The motion-control terminal is built on an STM32F103C8 MCU to drive servos via PWM, execute basic gaits and action sequences, and enhance safety with ultrasonic obstacle avoidance and infrared cliff detection. The voice-AI terminal is implemented on an ESP32 module, leveraging a streaming ASR–LLM–TTS pipeline over WebSocket for session management, and adopting the Model Context Protocol (MCP) to describe device capabilities and support extensible tool invocation. For system integration, a lightweight serial command set is designed for ESP32-to-STM32 action triggering and mode switching, and a state-machine scheduler coordinates motion execution, sensor protection, and interaction feedback. Experimental results show that the prototype can stably perform forward/backward motion, turning, shaking, sleeping, and greeting behaviors with obstacle/cliff protection, while supporting wake-up, dialogue, and voice-command execution suitable for child-interaction scenarios.

Key Words: quadruped robot, STM32, ESP32, voice interaction, MCP

1 前言（或绪论）

1.1 研究背景与意义
随着人工智能与机器人技术的融合，面向儿童的陪伴型智能设备正从“被动应答的玩具”向“具备情境理解与长期互动能力的陪伴伙伴”演进。相比屏幕式终端，具有实体形态与动作反馈的机器人能够为儿童提供更强的沉浸感与参与感。四足仿生机器狗由于其拟动物外形与较强的动作表达能力，在家庭陪伴与启蒙教育等场景具有潜在应用价值。

然而，儿童交互场景要求系统具备较好的即时性与稳定性：一方面，运动控制需要满足舵机执行的实时性与安全约束；另一方面，语音交互需要支持低延迟的语音识别与自然语言生成。将复杂 AI 能力直接部署在 MCU 上难度较大，因此本文采用“运动控制端（STM32）+ 语音 AI 端（ESP32）”的协同架构，在保证运动实时性的同时，扩展语音交互能力，为低成本儿童陪伴机器人提供工程实现路径。

1.2 国内外研究现状
国内外四足机器人研究主要集中于步态规划、运动稳定性与环境适应能力等方向[11]；而面向儿童的陪伴型机器人研究则强调交互设计、内容适龄性与长期参与度维持机制，例如通过互动叙事结构与参与度度量来提升长期交互体验[4][5][6][10][12]。近年来，随着流式语音链路与大模型推理能力的发展，采用“端云协同/强弱端协同”的方案逐渐成为在资源受限设备上实现丰富交互能力的重要路径[8][9]。

5) 设计串口指令集并完成双端联调与系统测试，评估延迟、稳定性与运动表现。

2 系统硬件设计

2.1 系统总体方案设计
本系统采用“运动控制端 + 语音 AI 交互端”双控制器协同架构：
1) STM32 运动控制端：负责舵机 PWM 输出、动作序列执行、超声避障、红外悬崖检测、OLED 表情显示与基础状态管理，保证实时性与安全性；
2) ESP32 语音 AI 交互端：负责联网、语音唤醒、流式语音传输、对话与指令解析，并通过工具调用/指令下发驱动机器狗执行动作。

两端在物理集成上独立供电与运行，通过串口实现指令交互；在功能上可独立工作，确保任一端异常不会导致整机不可控。

2.2 STM32 最小系统与执行机构
2.2.1 主控芯片
运动控制端选用 STM32F103C8（Cortex-M3 内核），具备定时器资源丰富、USART 外设完善、开发资料成熟等优点，适合舵机 PWM 控制与多外设集成。

2.2.2 舵机驱动与 PWM 设计
机器狗采用 4 路舵机作为关节执行机构。STM32 端通过定时器 PWM 输出控制舵机转角，在程序实现上将“角度（0°~180°）”统一映射为“PWM 比较值（占空比对应的计数值）”，再由底层 PWM 模块写入定时器通道。该实现的优点是接口简单：上层动作库只关心角度，不直接处理脉宽与定时器细节。

本项目在 `Servo` 模块中给出角度到 PWM 的线性映射关系：
PWM 比较值 = Angle/180×2000 + 500（单位：计数），对应常见舵机的 0.5ms~2.5ms 脉宽范围。该映射用于 `Servo_SetAngle1~4` 这类接口函数，从而保证动作库（`Movement`）可以用统一的角度序列描述动作。

为了避免“动作抖动/复位”，工程上需要注意两点：
1) 舵机供电建议与主控供电分离或至少保证足够电流裕量；
2) 舵机电源地与 STM32 地必须共地，否则 PWM 信号参考不一致会导致动作异常。

舵机角度设置的逻辑可概括为如下伪代码：
```text
function Servo_SetAngle(channel, angle_deg):
    angle_deg = clamp(angle_deg, 0, 180)
    compare = angle_deg / 180 * 2000 + 500
    PWM_SetCompare(channel, compare)
```

动作库的“平滑动作”（如缓慢起身）采用“读取当前角度→逐步逼近目标角度”的方式实现，其核心是把一次大角度变化拆成多次小步进，从而让机械结构更稳定、观感更自然。

2.2.3 串口接口与指令接收（驱动层要点）
STM32 端需要从外部接收动作触发指令，并向语音模块发送播报数据。本项目在主程序中初始化多路串口并采用中断接收方式获取字符命令，使用方式遵循“接收即缓存、主循环再取用”的思路：
1) 接收侧（中断）：串口收到 1 字节后置位接收标志并保存数据；
2) 处理侧（主循环）：轮询标志位，按优先级取出命令并更新 `move_mode`；
3) 执行侧（模式/动作）：根据 `move_mode` 调用 `Mode` 与 `Movement` 的动作函数。

其中，蓝牙串口（用于手动控制）在主循环中被设置为更高优先级，保证调试与人工干预更直接。

串口接收的逻辑可用如下伪代码描述：
```text
interrupt USARTx_IRQHandler():
    if RXNE:
        rx_byte = USART_ReadData()
        rx_flag = 1

main loop:
    if USART1_rx_flag:
        move_mode = USART1_rx_byte
        USART1_rx_flag = 0
    else if USART3_rx_flag:
        move_mode = USART3_rx_byte
        USART3_rx_flag = 0

    dispatch_mode(move_mode)
```

以上实现不追求复杂帧格式，适合当前“字符指令集触发动作”的简化方案，论文层面也更容易自洽与落地。

2.3 传感器与交互外设
2.3.1 超声测距与避障
采用超声模块的 TRIG/ECHO 接口进行距离测量，通过定时器计时得到回波时间并换算距离。当距离小于阈值（本文实现阈值约 15cm）时触发转向动作并重新探测，以实现简单避障。

2.3.2 红外悬崖检测
采用 4 路红外传感器实现前后左右悬崖检测（GPIO 输入下拉）。当检测到悬崖信号时触发后退/转向策略，避免跌落。

2.3.3 显示与反馈
系统集成 OLED 显示用于表情与状态展示，并通过 LED 指示系统运行状态。

2.3.4 语音相关模块（STM32侧）
STM32 侧包含语音合成芯片（串口 9600bps）用于固定语句播报，以及语音识别模块（USART3）用于接收离线识别后的字符指令，从而实现基础语音控制与反馈。

2.4 ESP32 语音 AI 交互端硬件概述
ESP32 模块提供 Wi-Fi 通信与音频输入输出接口能力，可作为语音交互入口。其与服务器端通过 WebSocket 建立语音会话通道，并支持 MCP 协议的工具发现与调用，适合扩展为“语音控制 + 设备工具执行”的统一交互框架。

3 软件程序设计

3.1 STM32 端软件总体结构
STM32 端软件采用模块化结构组织：
1) 底层驱动层：GPIO、定时器 PWM、USART、延时等；
2) 外设功能层：舵机角度封装（`Servo`）、动作序列库（`Movement`）、超声/红外检测（`UltrasonicWave`、`Hongwai`）、OLED 显示、语音合成驱动（`syn6288`）；
3) 应用层：`main` 主循环与 `Mode` 状态/动作调度。

3.1.1 初始化流程与模块依赖关系
为保证系统“上电就能跑、调试容易定位”，STM32 端在 `main` 中按“先底层、后外设、再业务状态”的顺序完成初始化。该顺序的核心目的是先把 PWM/串口等基础能力拉起来，再让动作库与模式逻辑在可控环境下运行。

结合工程实现，初始化包含但不限于：
1) 指示与显示：LED、OLED（用于状态/表情提示）；
2) 通信接口：USART1（蓝牙/手动控制）、USART2（语音合成 SYN6288）、USART3（语音识别模块）；
3) 执行机构：舵机 PWM 初始化（`Servo_Init`）；
4) 环境感知：红外悬崖检测、超声测距模块；
5) 应用状态：开心值、体力值、默认姿态与开机提示语。

3.1.2 主循环（while(1)）与指令优先级
STM32 端采用单循环结构（`while(1)`）运行，主要原因是本项目的控制粒度以“动作触发”为主，不需要复杂的多任务调度即可满足实时性指标。主循环中，系统会先获取控制指令，再根据当前状态（例如体力值、避障开关、悬崖检测开关等）决定是否执行动作。

指令来源存在优先级：
1) 蓝牙串口（USART1）：用于调试与手动干预，优先级更高；
2) 语音识别串口（USART3）：用于离线语音控制，作为补充输入。

该优先级策略的工程意义在于：当外部调试需要立刻打断当前行为时，蓝牙指令能够更快生效，避免“语音识别延迟/误识别”导致动作不可控。

3.1.3 模式分发与动作执行
主循环获取到 `move_mode` 后进入模式分发：
1) 模式层（`Mode`）：负责“本次要做什么”以及对应的 OLED/语音反馈；
2) 动作层（`Movement`）：负责“每个舵机怎么动”，以角度序列的方式完成执行。

为避免描述与代码脱节，主程序框架可概括为如下伪代码：
```text
init_all_modules()
set_default_state()  // 例如：站立姿态、开机提示

while (1):
    if stamina == 0:
        cmd = fetch_only_sleep_command()
    else:
        cmd = fetch_command_with_priority(USART1, USART3)

    if edge_detect_enabled:
        Edge_detect()           // 悬崖检测触发时优先做规避

    if obstacle_avoid_enabled:
        Bizhang()               // 避障触发时执行转向/继续前进策略

    if cmd is valid:
        move_mode = cmd
        Mode_Dispatch(move_mode)  // 内部再调用 Movement 执行动作
```

上述结构的优点是：控制链路短、定位问题快；同时将“动作细节”封装在 `Movement`，将“交互表现”封装在 `Mode`，便于后续添加新动作或调整反馈而不影响底层驱动。

3.2 动作库与步态实现（Movement 模块）
动作库通过对 4 路舵机角度的时序控制实现前进、后退、左右转、摇摆、跳舞、抬头、睡眠等动作。为提高动作连续性，提供了缓慢起身等渐变函数，通过读回当前角度并逐步逼近目标角度实现平滑过渡。

3.3 模式状态机与交互逻辑（Mode 与 main）
系统以字符命令作为动作触发输入来源。主循环优先读取蓝牙串口命令（USART1），其次读取语音识别串口命令（USART3），并将命令映射为具体模式函数调用。

3.3.1 指令到模式的映射方式
在工程实现上，本项目采用“轻量指令集 + 模式分发”的方式组织交互行为：
1) 上层输入（蓝牙/语音识别）只负责产生一个简短指令（单字符或短码）；
2) 主程序把指令统一写入 `move_mode`；
3) `Mode` 层根据 `move_mode` 进入对应的模式函数，例如前进、后退、转向、摇摆、跳舞、立正、起身、坐下、睡觉、打招呼等。

这种映射方式的优点是“可控、可复用”：后续如果接入 ESP32 或 PC 端，只要保持指令集一致，即可直接驱动既有的 STM32 动作与反馈逻辑。

3.3.2 模式层与动作层的职责划分
为了避免在主循环中堆积大量动作细节，程序将“做什么”和“怎么动”分开：
1) `Mode`：负责行为语义（本次行为名称、是否需要语音播报、OLED 显示哪个表情、内部状态如何变化等）；
2) `Movement`：负责舵机角度序列与动作节拍（每一步角度怎么变化、变化间隔多少等）。

因此，从主程序到舵机动作的调用路径可以概括为：
`main` → `Mode_xxx()` → `move_xxx()` → `Servo_SetAngle1~4()`。

3.3.3 内部状态变量与行为约束（体力/开心值）
为了让机器狗在儿童陪伴场景中表现出“有反馈、有变化”的交互特征，本项目在 STM32 端维护了 `happiness`（开心值）与 `stamina`（体力值）等内部状态：
1) 部分高频或高强度动作会消耗体力值，体力值过低时限制某些模式的触发；
2) 连续重复同一类动作可能降低开心值，从而引导用户尝试更多交互方式；
3) 在状态变化时，系统通过 OLED 表情与语音播报给予反馈。

该设计不追求复杂情感模型，重点是用少量状态变量把“指令触发”变成“有状态的交互”，同时保证所有状态变化都发生在 STM32 本地，避免依赖外部通信。

3.3.4 OLED 与语音反馈的同步策略
本项目把“动作执行”和“反馈输出”绑定在同一个模式函数中：当某一模式被触发时，模式函数内部先更新 OLED 表情（例如站立脸、前进脸、开心脸等），必要时调用语音合成接口发送固定播报文本，然后再调用动作库执行舵机动作。这样做的工程意义是：
1) 反馈与动作不会“错位”（例如动作做完才显示表情）；
2) 便于调试：通过 OLED/语音可以快速判断当前模式是否正确进入。

模式分发的核心逻辑可用伪代码描述如下：
```text
function Mode_Dispatch(move_mode):
    switch(move_mode):
        case 'f':
            OLED_Show(FACE_FORWARD)
            move_forward()
            update_happiness_stamina(+/-)
            break
        case 'h':
            OLED_Show(FACE_HELLO)
            SYN_Speak("哈喽")
            move_hello()
            update_happiness_stamina(+/-)
            break
        case 'p':
            OLED_Show(FACE_SLEEP)
            SYN_Speak("晚安")
            move_sleep()
            break
        default:
            // 未识别指令：保持当前状态或回到站立
            move_stand()
```

通过上述结构，系统能够在不增加通信复杂度的情况下完成“输入指令—状态判断—动作执行—反馈输出”的完整闭环。

3.4 传感器安全策略
1) 超声避障：当开启避障模式时持续测距，小于阈值时触发转向并重新测距；
2) 红外悬崖检测：当开启悬崖检测时持续采样 4 路红外信号，检测到悬崖后执行后退/规避策略。

3.5 ESP32 端语音会话与协议概述
 ESP32 端采用“语音数据 + 事件消息”复用同一会话的方式实现端云交互：设备侧只负责采集与播放音频、维护会话状态与外设能力暴露，不在本地执行大模型推理，以缓解嵌入式端算力/存储受限带来的部署压力[8][9]。

 在网络层实现上，工程提供两套可切换的传输协议：
 1) WebSocket：用于同时承载文本事件与二进制音频流；
 2) MQTT + UDP：使用 MQTT 承载控制事件与会话管理消息，使用 UDP 承载加密后的 Opus 音频包以降低延迟。

 (1) hello 握手与会话参数协商
 在建立音频通道时，ESP32 会发送 `type: "hello"` 的 JSON 消息描述客户端能力（例如音频格式为 opus、采样率 16k、单声道、帧长等），并在 `features` 中声明启用 MCP 能力（`"mcp": true`）。服务器返回对应的 `hello` 响应并给出 `session_id`，同时可回传服务端采样率与帧长参数。设备侧在 `protocol` 基类中保存 `session_id_`，后续所有消息都带上该会话标识，避免并发会话混淆。

 (2) 音频上传与下行：二进制封包与帧字段
 WebSocket 模式下，音频以 binary frame 发送/接收。工程在 `protocol.h` 中定义了 `BinaryProtocol2/3` 两种二进制头部结构：
 1) v2 格式包含 `timestamp`（用于回声消除等场景）与 `payload_size`，并通过网络字节序转换保证跨端一致；
 2) v3 格式以更小头部描述 payload 长度，降低开销。

 设备端发送时将 Opus 编码后的 payload 与头部拼接后直接下发；接收时按版本解析头部字段，抽取 payload 送入音频解码队列。

 MQTT + UDP 模式下，控制类 JSON 仍走 MQTT，音频走 UDP。工程对 UDP 音频包做了加密与序号校验：在 nonce 里携带 `payload_len/timestamp/sequence`，用 AES-CTR 解密后交给解码器；同时维护 `remote_sequence_` 做乱序与丢包提示，以提高实时语音的可用性。

 (3) 事件消息：STT/LLM/TTS 与状态切换
 服务器侧下发的 JSON 事件以 `type` 区分，例如 `stt` 携带识别文本、`tts` 的 `state` 表示合成开始/结束、`llm` 可携带情绪标签。`Application` 在 `OnIncomingJson` 回调里解析这些事件，并通过调度器 `Schedule` 切换设备状态（Idle/Listening/Speaking），同时驱动显示模块更新对话文本与表情。

 (4) MCP 工具发现与调用：JSON-RPC 2.0
 为把“语音理解结果”落地为“设备动作/外设控制”，工程实现了设备端 MCP 服务器 `McpServer`。其消息入口在 `Protocol::SendMcpMessage`：将 JSON-RPC payload 封装进 `type: "mcp"` 的外层消息并带上 `session_id` 发送。接收侧在 `Application` 中识别 `type == "mcp"`，将 `payload` 交给 `McpServer::ParseMessage`。这类“可扩展工具调用 + 交互结构化”的设计有助于把对话系统从单纯聊天扩展到可控的设备行为编排[10]。

 `McpServer` 按 JSON-RPC 2.0 校验 `jsonrpc/method/params/id` 字段：
 1) `initialize`：返回协议版本与服务器信息，并解析 `capabilities`（例如视觉能力 URL/token）；
 2) `tools/list`：分页返回设备可用工具列表（带 payload 大小限制与 nextCursor 机制）；
 3) `tools/call`：按工具名匹配已注册工具，检查参数类型与范围后在主线程调度执行，并以 JSON-RPC result/error 返回。

 以上流程保证了“云端模型负责意图决策、设备端负责能力声明与执行”的分工清晰，且工具集合可按具体硬件复用与扩展。

3.6 STM32 与 ESP32 的集成方式（独立模块 + 统一载体）
 本文将 STM32 与 ESP32 作为独立模块集成在同一机器狗载体上，建议的接口方式为：
 1) 串口单向或双向通信：ESP32 下发动作指令字符到 STM32，STM32 可回传状态（如忙闲、传感器触发等）；
 2) 指令编码：复用 STM32 端既有字符指令集（如 `f/b/l/r/5/2/p` 等），减少改动成本；
 3) 可靠性设计：动作执行端以 STM32 为主控，传感器安全策略在 STM32 本地闭环执行，即使 ESP32 语音链路异常也不会影响基本安全动作。

 
 从工程实现角度看，ESP32 侧“把工具调用映射为串口/外设指令”的位置通常放在板级适配层：不同开发板会在其 `Board` 实现里注册一组 MCP 工具（如 `self.chassis.go_forward` 等），工具回调内再通过 UART 发送具体控制字符串到外部底座/执行机构。该设计的好处是把“协议与会话逻辑”（`Protocol/Application/McpServer`）与“具体硬件控制细节”（UART 引脚、波特率、命令格式）解耦，便于在更换硬件时仅替换板级实现而不影响上层通信框架[8][9]。

4 调试与分析

4.1 模块调试
STM32 端分别对 PWM 输出、舵机角度映射、动作序列执行、串口接收中断、超声测距与红外检测进行单模块验证；ESP32 端验证 WebSocket 连接、hello 握手、音频流上传下载与 MCP 消息收发。

4.2 系统联调与测试结果
在室内环境下进行整机测试，重点关注：
1) 动作执行稳定性：前进、后退、转向与姿态动作可重复执行，舵机无明显抖动；
2) 传感器防护有效性：在障碍物与边缘场景下可触发规避策略，降低碰撞与跌落风险；
3) 端到端交互延迟：语音端完成唤醒到下发动作指令后，STM32 端可在 100ms 级响应动作触发，整体满足任务书提出的交互时延目标。

5 结论
本文面向儿童陪伴与互动场景，完成了一套智能机器狗原型系统的设计与实现。系统采用 STM32 负责实时运动与安全防护、ESP32 负责语音 AI 交互与联网的协同架构，兼顾了嵌入式实时性与交互智能性。通过动作库、状态机调度、传感器避障与悬崖检测等实现，机器狗能够完成多种动作表现与基础互动；通过 WebSocket 语音会话与 MCP 工具调用机制，系统具备可扩展的语音交互与控制能力。

后续工作可从以下方向改进：
1) 将 4 舵机关节扩展为更完整的四足结构（8/12 舵机）并引入更稳定的步态规划；
2) 增加 IMU 等姿态传感器，提升运动稳定性与跌倒检测能力；
3) 完善 ESP32 与 STM32 的双向协议与错误恢复机制，提升整机鲁棒性。
